# Restate v1.7.0 Release Notes

## Highlights

- **Flow Control**: Restate begins shipping its flow-control primitives. v1.7 introduces the first building block — **concurrency limits** driven by a request *scope*, exposed through new `/restate/` ingress endpoints. More flow-control capabilities will follow in later releases.

- **Native Google Cloud Run Authentication**: HTTP deployments can now opt into native Google OIDC ID-token authentication. Restate mints short-lived, audience-scoped, Google-signed tokens for every discovery and invocation request, replacing the pattern of embedding long-lived bearer tokens in deployment headers.

- **Dynamic Kafka Cluster Management**: Kafka clusters can now be configured dynamically through the Admin API, and CLI, instead of being baked into static `restate.toml` configuration. Subscriptions and clusters are both manageable from the `restate` CLI.

- **Bounded Invoker Memory Pool**: The invoker now runs under a bounded memory pool that limits in-flight data flowing from the server to deployments. This adds backpressure and protects nodes from out-of-memory conditions under bursts of large payloads or state, making the system more predictable under load.

- **Restate UI 1.0**: The bundled Restate web UI graduates from `0.x` to **v1.0** — it comes with a complete new layout, better observability features and much more. The UI ships with every server release, so the upgrade lands automatically with no configuration.

## Breaking Changes

### Per-Database RocksDB Background Work Budgets

The `rocksdb-max-background-jobs` configuration option has been removed from the shared `rocksdb` config block (which was applied to every database identically). Background work is now controlled per-database with separate flush and compaction concurrency limits that are automatically computed based on the node's CPU count and active roles.

Each database type now has its own `rocksdb-max-background-flushes` and `rocksdb-max-background-compactions` options in its respective config section. A new `export-concurrency-limit` option has been added to the `[worker.snapshots]` section to control the number of concurrent snapshot exports (previously derived from `rocksdb-max-background-jobs`).

Previously, every database on a node (partition-store, log-server, metadata-server, local-loglet) was independently allocated the full CPU count as its background job budget, leading to up to `4 * CPU_COUNT` background jobs competing for the same shared thread pool. The new approach splits background work into two dimensions: **flushes** (latency-critical) are allocated equally across databases, while **compactions** (throughput-heavy) are weighted toward the partition-store (~65%). When a role is not active on the node, its budget is redistributed to the remaining databases.

**Impact on Users**:
- **Deployments using `rocksdb-max-background-jobs`**: This key is no longer recognized. Migrate to the new per-database options.
- **Deployments using defaults**: No action needed; the new defaults are computed automatically.

**Migration Guidance**:

```toml
# Old (no longer recognized)
[worker.storage]
rocksdb-max-background-jobs = 8

# New: per-database control
[worker.storage]
rocksdb-max-background-flushes = 4
rocksdb-max-background-compactions = 8

[log-server]
rocksdb-max-background-flushes = 4
rocksdb-max-background-compactions = 4

[metadata-server]
rocksdb-max-background-flushes = 1
rocksdb-max-background-compactions = 1
```

For snapshot export concurrency (previously derived from background jobs):

```toml
[worker.snapshots]
export-concurrency-limit = 4
```

#### Renamed Global RocksDB Thread-Pool Options

Two `[common]` options that size the shared RocksDB background thread pools have been renamed:

| Old key | New key |
| --- | --- |
| `rocksdb-bg-threads` | `rocksdb-low-priority-threads` |
| `rocksdb-high-priority-bg-threads` | `rocksdb-high-priority-threads` |

Their defaults also changed. `rocksdb-high-priority-threads` (flush pool) now defaults to `ceil(CPU / 4)`, and `rocksdb-low-priority-threads` (compaction pool) defaults to the CPU cores left over after the high-priority reservation. These two pools now also drive how the per-database flush and compaction budgets above are split between the partition-store, log-server, and metadata-server.

```toml
# Before
[common]
rocksdb-bg-threads = 8
rocksdb-high-priority-bg-threads = 4

# After
[common]
rocksdb-low-priority-threads = 8
rocksdb-high-priority-threads = 4
```

#### Removed Log-Server RocksDB Options

Two `[log-server]` options have been removed:

- `rocksdb-max-wal-size` — the maximum total WAL size is now derived internally from the write-buffer size and the maximum number of write buffers.
- `rocksdb-max-background-flushes` — the log-server now uses a fixed, dedicated flush budget. Its data column family flushes atomically with a single flushing CF, so it can never have more than one flush in flight.

Remove these keys if present; they are no longer recognized:

```toml
[log-server]
# rocksdb-max-wal-size = "..."          # remove
# rocksdb-max-background-flushes = ...  # remove
```

### Kafka and Shuffler Batch Ingestion Now Mandatory

The batch ingestion path that was introduced as opt-in experimental in v1.6 is now the only implementation for both Kafka subscriptions ingestion (`restate-ingress-kafka`) and partition-to-partition shuffle (the partition processor's outbox shuffle). The two opt-in flags that previously gated this behavior are removed:

- `experimental-kafka-batch-ingestion`
- `experimental-shuffler-batch-ingestion`

The corresponding legacy code paths (the bifrost-based shuffle state machine and the legacy Kafka consumer/dispatcher) have been deleted. The batch ingestion path leverages the ingestion client to amortize append work across multiple records, giving substantially higher throughput on Kafka ingress and on cross-partition message shuffling.

**Impact on Users**:
- **Users who never set either flag** (the vast majority): no action required; you pick up the faster path automatically.
- **Users who set either flag to `true`**: no behavior change; remove the now-unrecognized keys.
- **Users who explicitly set either flag to `false`** to stay on the legacy path: this preference is no longer honored — the legacy implementation has been removed and there is no equivalent option.

**Migration Guidance**: Remove these keys from any `restate.toml` or environment overrides if present:

```toml
experimental-kafka-batch-ingestion = ...        # remove
experimental-shuffler-batch-ingestion = ...     # remove
```

Equivalent env vars to drop: `RESTATE_EXPERIMENTAL_KAFKA_BATCH_INGESTION`, `RESTATE_EXPERIMENTAL_SHUFFLER_BATCH_INGESTION`. No data migration is required.

### Snapshot Retention Graduated Out of Experimental

The opt-in snapshot retention feature introduced in v1.6 is now a stable, always-on part of the snapshot repository:

1. **Config option renamed and made non-optional.** `worker.snapshots.experimental-num-retained` (an optional integer, disabled by default) has been replaced by `worker.snapshots.num-retained`, a non-optional integer with a default of `1`.
2. **`latest.json` is always written in V2 format.** Every snapshot upload writes a V2 `latest.json` pointer.
3. **Legacy V1 pointers continue to be readable** and are upgraded to V2 the next time a snapshot is uploaded for that partition — no operator action is required on existing repositories.

Restate is now responsible for cleaning up snapshots that are no longer used for bootstrapping partition processors (because the log has been trimmed beyond their archived LSN), preventing unbounded snapshot storage growth.

**Impact on Users**:
- **Existing repositories**: no migration required. The first upload after the upgrade rewrites `latest.json` from V1 to V2 in place. Snapshot data files are unchanged.
- **Existing configurations**: any deployment that explicitly set `experimental-num-retained` (in `restate.toml` or via `RESTATE_WORKER__SNAPSHOTS__EXPERIMENTAL_NUM_RETAINED`) **must rename the key to `num-retained`** before upgrading. The old key is ignored.
- **Default behavior**: `num-retained = 1` keeps exactly one snapshot per partition and cleans up older snapshots that are no longer usable due to log trimming.
- **Downgrade**: rolling back to a binary older than v1.6 is not supported once V2 pointers have been written. Rolling back to v1.6.x is safe.

**Migration Guidance**:

```toml
# Before
[worker.snapshots]
experimental-num-retained = 3

# After
[worker.snapshots]
num-retained = 3
```

_Related: #4155, #3942 (original experimental introduction in v1.6)_

### Ingress Request Size Limit Enforced

The HTTP ingress now enforces a maximum request body size. Requests whose body exceeds the limit are rejected with `413 Payload Too Large` before reaching the handler. A new configuration option `ingress.request-size-limit` controls the cap. If unset, it defaults to `networking.message-size-limit` (32 MiB by default). If set, the value is always clamped at `networking.message-size-limit`, since larger requests cannot be transmitted over the cluster-internal network.

Previously, the ingress accepted request bodies of unbounded size and only failed later — often deep in the pipeline — once cluster-internal size limits were hit. Oversized requests are now rejected early with a standard 413 response.

**Impact on Users**:
- **Default behavior changes**: requests larger than `networking.message-size-limit` (32 MiB by default) are now rejected with 413 instead of being accepted and failing later. Clients that relied on submitting larger payloads must either reduce request size or raise `networking.message-size-limit`.
- **Existing deployments**: review whether any clients send bodies larger than 32 MiB.

**Migration Guidance**: To raise the ingress limit, raise the networking limit as well:

```toml
[networking]
message-size-limit = "64MB"

[ingress]
request-size-limit = "64MB"
```

Setting `ingress.request-size-limit` higher than `networking.message-size-limit` has no effect — the value is clamped to the networking limit.

### Default Bind Address Now Dual-Stack

The default `bind-ip` has changed from `0.0.0.0` (IPv4 only) to `::` (IPv6 unspecified), which on most systems creates a dual-stack socket that accepts both IPv4 and IPv6 connections. Additionally, automatic routable IP detection now prefers IPv6, falling back to IPv4 when IPv6 is not available.

In IPv6-only environments (e.g. Kubernetes clusters with IPv6-only networking), Restate previously failed to start because `0.0.0.0` cannot bind when no IPv4 stack is available. By defaulting to `::`, the server now listens on both IPv4 and IPv6 interfaces.

**Impact on Users**:
- **Existing IPv4 deployments**: No action required. On Linux and macOS, `::` with the default `IPV6_V6ONLY=false` kernel setting creates a dual-stack socket that continues to accept IPv4 connections via IPv4-mapped IPv6 addresses.
- **IPv6-only deployments**: Restate now starts out of the box without needing to configure `bind-ip`.
- **Explicit `bind-ip` configured**: this change has no effect — your explicit value takes precedence.
- **Edge case**: On the rare Linux systems where `net.ipv6.bindv6only=1` is set, binding to `::` will only accept IPv6 connections. Set `bind-ip = "0.0.0.0"` to restore IPv4-only behavior.

**Migration Guidance**: No migration is required for the vast majority of deployments. To restore the previous behavior:

```toml
bind-ip = "0.0.0.0"
```

Or via `RESTATE_BIND_IP=0.0.0.0`.

_Related: #4529_

### Bifrost Retry Policies Replaced by Adaptive Store Timeouts

The sequencer now uses adaptive timeouts when sending store waves to log servers, replacing the previous fixed-timeout + exponential-backoff retry policy. The adaptive algorithm tracks per-node store latencies in a sliding window and derives the wave timeout from the P99.99 of recent observations, so the timeout automatically tightens when nodes are fast and relaxes when they are slow — without manual tuning.

The `rpc_timeout` configuration field under `[bifrost.replicated-loglet]` has changed type from a fixed `Duration` to a `BackoffInterval`, expressed as a compact duration-range string `"<min>..<max>"`. The default is `"250ms..60s"`. It now governs adaptive timeouts (and retry delays) for general log-server RPC operations.

The timeout for the store waves themselves is configured separately by the new `store_timeout` field (also a `BackoffInterval`), defaulting to `"2s..60s"`. Splitting it out from `rpc_timeout` lets the store path use a more relaxed range suited to the current spread selector, independent of the tighter RPC timeout. Most deployments need not set it — the default is sized for typical replicated-loglet workloads.

The old `sequencer_retry_policy` and `log_server_retry_policy` fields are ignored if present and should be removed from your configuration.

**Impact on Users**:
- **Existing deployments**: automatically benefit from adaptive timeouts on upgrade. No action required unless you customised `sequencer_retry_policy` or `log_server_retry_policy`.
- **Customised retry policy**: migrate to `rpc_timeout` (see below). The new field covers both the per-wave timeout and the inter-wave backoff delay.

**Migration Guidance**:

```toml
# Before (deprecated, will be ignored)
[bifrost.replicated-loglet]
sequencer_retry_policy = { type = "exponential", initial_delay = "250ms", max_delay = "5s" }
log_server_retry_policy = { type = "exponential", initial_delay = "250ms", max_delay = "2s" }

# After
[bifrost.replicated-loglet]
rpc_timeout = "250ms..15s"
# Optional: override the store-wave timeout range (default "2s..60s")
store_timeout = "2s..30s"
```

The duration-range format accepts the same unit designators as the rest of Restate's configuration (`ms`, `s`, `m`, `h`, `d`, and verbose forms). Fractional values such as `0.5s` are also accepted.

## Deprecations

### `disable_eager_state` Deprecated

The private config option `disable_eager_state` is deprecated in favor of `eager-state-size-limit = "0"`. Existing configs using `disable_eager_state: true` continue to work and are internally treated as `eager-state-size-limit = "0"`. See [Eager State Size Limit](#eager-state-size-limit) for the replacement.

### Root-Level Service-Client Options Moved Under `worker.invoker`

The service-client configuration options — HTTP client tuning, AWS Lambda options, request identity, and additional outbound headers — have moved from the TOML root to a dedicated section under `worker.invoker`. The root-level location continues to work but emits a deprecation warning, and will be removed in Restate v1.8.

Affected keys (previously at the TOML root):

- HTTP client: `http-keep-alive-options.interval`, `http-keep-alive-options.timeout`, `http-keep-alive-options.jitter`, `http-proxy`, `no-proxy`, `connect-timeout`, `initial-max-send-streams`, `streams-per-connection-limit`, `idle-connection-timeout`, `http2-initial-stream-window-size`, `http2-initial-connection-window-size`, `http2-max-frame-size`
- AWS Lambda: `aws-profile`, `aws-assume-role-external-id`, `request-compression-threshold`
- Identity / headers: `request-identity-private-key-pem-file`, `additional-request-headers`

**Impact on Users**:
- **Existing configs**: any affected key at the TOML root is still read on startup, with a deprecation warning telling you to move it.
- **Conflicts**: if both locations set the same key, the deprecated (root) value takes precedence and a warning is emitted — remove the deprecated key once you've moved it.
- **Removal**: the root-level keys are removed in Restate v1.8; migrate during the v1.7 window.

**Migration Guidance**:

```toml
# Before — at the TOML root
http-proxy = "http://proxy.internal:3128"
no-proxy = "localhost,127.0.0.1"
aws-profile = "restate-prod"
request-identity-private-key-pem-file = "/var/secrets/key.pem"

[http-keep-alive-options]
interval = "1m"
timeout = "30s"
```

becomes:

```toml
# After — under worker.invoker
[worker.invoker]
http-proxy = "http://proxy.internal:3128"
no-proxy = "localhost,127.0.0.1"
aws-profile = "restate-prod"
request-identity-private-key-pem-file = "/var/secrets/key.pem"

[worker.invoker.http-keep-alive-options]
interval = "1m"
timeout = "30s"
```

## New Features

### Flow Control

v1.7 ships the first building block of Restate's **flow control** primitives: **concurrency limits** driven by a request *scope*. A scope acts as a namespace for virtual-queue–based concurrency control. More flow-control capabilities will follow in later releases.

**Why flow control matters.** As soon as many invocations compete for the same downstream resources, you need a way to shape that traffic rather than let it run unbounded. Flow control gives you that lever:

- **Cost control** — cap how much expensive work runs at once. This is especially valuable for AI agents, where each concurrent invocation may translate directly into model/API spend; a concurrency limit per scope puts a ceiling on that cost.
- **Endpoint protection** — keep a burst of invocations from overwhelming a downstream service, database, or third-party API by bounding how many hit it concurrently.
- **Fairness** — because invocations now flow through a scheduler that decides who goes next, the system ensures fairness between invocations running on the same partition.

Concurrency limits are the first capability built on this scheduler. Planned follow-ups include throttling and rate limits, invocation priorities, finite queue (backlog) limits, and more — all expressed through the same scope-based model introduced here.

**Enabling flow control**:

```toml
experimental_enable_vqueues = true
```

Or via environment variables: `RESTATE_EXPERIMENTAL_ENABLE_VQUEUES=true`.

#### New ingress endpoints

The ingress HTTP server gained a set of endpoints under the reserved `/restate/` prefix that carry scope for concurrency control, plus endpoints to attach to or look up running invocations:

```
# Non-scoped service calls
POST /restate/call/{service}/{handler}
POST /restate/call/{service}/{key}/{handler}
POST /restate/send/{service}/{handler}
POST /restate/send/{service}/{key}/{handler}

# Scoped service calls (for concurrency control)
POST /restate/scope/{scopeKey}/call/{service}/{handler}
POST /restate/scope/{scopeKey}/call/{service}/{key}/{handler}
POST /restate/scope/{scopeKey}/send/{service}/{handler}
POST /restate/scope/{scopeKey}/send/{service}/{key}/{handler}

# Attach to / read the output of an invocation by its id
GET /restate/attach/{invocation_id}
GET /restate/output/{invocation_id}

# Or attach to / read the output using a target address (workflow, idempotent invocation, or invocation id)
POST /restate/attach
{ "target": "workflow", "workflowName": "MyWorkflow", "workflowKey": "wf-1", "scope": "optionalScope" }

POST /restate/attach
{ "target": "idempotentInvocation", "service": "MyService", "key": "optionalObjectKey", "handler": "myHandler", "idempotencyKey": "my-key", "scope": "optionalScope" }

POST /restate/attach
{ "target": "invocation", "invocationId": "inv_1abc..." }

# Same for /restate/output

# You can also use the same target address to resolve the invocation id of an invocation using lookup
POST /restate/lookup
{ "target": "workflow", "workflowName": "MyWorkflow", "workflowKey": "wf-1", "scope": "optionalScope" }

POST /restate/lookup
{ "target": "idempotentInvocation", "service": "MyService", "key": "optionalObjectKey", "handler": "myHandler", "idempotencyKey": "my-key", "scope": "optionalScope" }
```

The old unversioned paths (`/{service}/{handler}`, `/restate/invocation/...`, `/restate/workflow/...`, etc.) continue to work unchanged but are to be consider deprecated, and will be removed in future releases.

#### Configuring concurrency limits (the rule book)

Concurrency limits are defined in a cluster-wide **rule book**. A rule pairs a *pattern* — which selects the `scope`s it applies to — with a set of *limits*. The only limit available in v1.7 is `concurrency`: the maximum number of invocations that may run concurrently for a matching scope. More limit kinds will follow in later releases.

A pattern is either an exact scope or the wildcard `*`:

- `*` — every scope (a global default)
- `myScope` — a specific scope

When both a wildcard and an exact-scope pattern match an invocation, the more specific (exact-scope) rule wins.

Rules are managed dynamically through the `restate` CLI. The `restate rules` commands let you list, create-or-update, disable, and delete rules. `set` is idempotent: on a new rule it creates the rule; on an existing rule it merges into the current values, preserving fields you don't touch. `set` (on an existing rule), `enable`, `disable`, and `delete` carry an optimistic-concurrency precondition under the hood, so two operators editing the same rule concurrently get an actionable error rather than a silent overwrite.

```bash
# Set a global default of 1000 concurrent invocations, plus a tighter
# 50-concurrency cap for the "checkout" scope.
restate rules set "*" --concurrency 1000 --description "global default"
restate rules set "checkout" --concurrency 50

# Inspect what's configured
restate rules list                # one row per rule
restate rules list --extra        # also shows description, version, last-modified

# Soft-disable or re-enable a rule without losing its definition
restate rules disable "checkout"
restate rules enable  "checkout"

# Remove the checkout rule again
restate rules delete "checkout"
```

Run `restate rules --help` for the full surface.

Send requests through the scoped ingress endpoints so they carry a `scope`; matching invocations are then throttled to the configured `concurrency` and held in their vqueue until a slot frees up. The applied rule book version per partition is visible as `applied_rule_book_version` in `partition_state` (and the `RULE-BOOK` column in `restatectl partition list`); the live rule set and per-counter usage are queryable via the `sys_rules` and `sys_user_limits` tables described below.

#### Limitation: enable vqueues only on fresh clusters

In v1.7, vqueues can only be enabled on fresh clusters — clusters that have no in-flight invocations. Migrating a cluster with existing in-flight invocations onto vqueues is not yet supported; that capability is planned for a following release.

Until migration support lands, enable `experimental_enable_vqueues` only on freshly provisioned clusters.
Attempting to enable vqueues on a cluster with in-flight invocations will result in the cluster not booting up.

#### Observing flow control

Five new SQL system tables let you inspect vqueues, the scheduler, and concurrency limits directly via the storage query interface:

- **`sys_rules`** — the configured rule book: one row per rule with its `pattern`, `concurrency` limit, `description`, `disabled` flag, `version`, and `last_modified` time.
- **`sys_vqueues`** — one row per entry across all vqueue stages (`inbox`, `running`, `paused`, `suspended`, `finished`), with the entry's `status`, `entry_kind` (`invocation` / `state-mutation`), attempt/suspension/yield counters, lifecycle timestamps, whether it holds a lock, and (for inbox entries) when it becomes runnable.
- **`sys_vqueue_meta`** — aggregate metadata and statistics per vqueue: its `scope`, `service_name`, active/paused flags, per-stage entry counts, and exponential moving averages of queue/run/suspension/end-to-end durations and time spent blocked on concurrency rules or invoker throttling.
- **`sys_scheduler`** — real-time scheduler state for each vqueue's head entry: queue depth, scheduler `status` (e.g. `Dormant`, `Ready`, `BlockedOn`, `Scheduled`), the `head_entry_id`, the `blocked_on` resource, and a per-resource breakdown of how long the head has been blocked (invoker concurrency, throttling rules, invoker throttling, invoker memory, concurrency rules, lock, deployment concurrency).
- **`sys_user_limits`** — per-scope concurrency-limit counters: current `usage`, configured `concurrency_limit`, `available` capacity, the matching `rule_pattern`, and the number of vqueues waiting on each counter. (Shape may change in future releases.)

These tables are populated only when vqueues is enabled.

#### Pausing invocations

When vqueues is enabled, the pause command behaves slightly differently:

- **It is persisted and applied by the partition processor**, so a pause is honored deterministically regardless of whether the invoker is currently running the invocation (previously a pause could be silently dropped if the invoker was not actively processing it).
- **Suspended invocations can now be paused** (in addition to running ones). Resuming re-invokes the invocation and replays its journal.
- **Graceful drains are no longer supported.** Pausing a running invocation immediately transitions it to `Paused` and aborts the in-progress execution attempt, instead of waiting for the invoker to reach a safe point. Journaled progress is preserved and replayed on resume; any in-flight, not-yet-journaled work of the aborted attempt is re-executed on resume, following the usual durable-execution replay semantics.

Invocations that are not on vqueues keep the previous best-effort pause behavior.

### Service Protocol v7

Restate v1.7 introduces **service protocol v7**, an opt-in iteration of the runtime↔SDK wire protocol that is negotiated with deployments advertising support for it. Older deployments continue to run on v6 or earlier with no change.

**Enabling service protocol v7**:

```toml
experimental_enable_protocol_v7 = true
```

Or via environment variable: `RESTATE_EXPERIMENTAL_ENABLE_PROTOCOL_V7=true`.

> **Note**: enabling protocol v7 is **one-way** — once a v7 invocation has been journalled, rolling back to a Restate server version < v1.7 is no longer supported.

**What you gain by enabling v7**:

- **Direct control over failure handling.** Services can mark an error to override the default retry-with-backoff behavior. An error can request that the invocation **pause** (stop and wait to be resumed by hand later — e.g. on an unrecoverable validation failure) or **fail** (terminate immediately without retrying — e.g. on an unrecoverable journal mismatch). Either way the handler avoids burning retry budget on a doomed invocation.
- **Lighter `ctx.run()` confirmations.** Acknowledging that a `ctx.run()` result has been persisted no longer requires the runtime to replay the full notification back to the SDK; a compact ack is sent instead. The result: less per-`ctx.run()` overhead, which matters for handlers that call `ctx.run()` frequently.
- **Better visibility into suspended invocations.** Suspension now records exactly what an invocation is blocked on (which completions, signals, or combinations of futures), and emits a `SuspendedEvent` into the journal events stream when it happens. That gives you a structured record in introspection / observability tooling instead of an opaque "suspended" status.
- **End-to-end flow-control context.** Scope, limit-key, and idempotency-key are now carried through the wire protocol — propagated to handlers, child calls, workflow targets, and idempotent targets. This is what lets SDKs participate in [Flow Control](#flow-control) and propagate the keys injected by [Safe Ingress Retries](#safe-ingress-retries) end-to-end through call graphs.

Which of these surface in your handlers depends on SDK support — consult your SDK's release notes for the v7 features it wires up.

### Native Google Cloud Run Authentication

HTTP deployments registered with Restate may now opt into native Google OIDC ID-token authentication. When configured, Restate mints a Google-signed ID token for every discovery and invocation request to the deployment and attaches it as `X-Serverless-Authorization: Bearer <token>`. Cloud Run validates this header in precedence over `Authorization` and strips it before forwarding the request to the container, so any `Authorization` value the operator placed in the deployment's `additional_headers` passes through to the workload unchanged.

Until now the only way to invoke a private Cloud Run service was to embed a long-lived bearer token in the deployment's `additional_headers` — a pattern with no rotation, no audit story, and which forced users to ship service-account keys to the Restate node. Native ID-token auth replaces it with short-lived, audience-scoped, Google-signed tokens the receiving service can verify against Google's JWKS.

Two configuration knobs are added to each HTTP deployment:

- `impersonate_service_account`: an SA email Restate impersonates via the IAM Credentials `generateIdToken` API. Defaults to the ambient ADC identity.
- `audience`: an explicit OIDC `aud` claim. Defaults to the deployment URL's origin (scheme + host + optional port).

The CLI gains three new flags on `restate dp register`:

```bash
# Cloud Run service callable by the ambient identity:
restate dp register https://svc-abc-uc.a.run.app --gcp-id-token

# Impersonate a target service account:
restate dp register https://svc-abc-uc.a.run.app \
  --gcp-impersonate-service-account caller@my-proj.iam.gserviceaccount.com

# Custom domain in front of Cloud Run; override the audience:
restate dp register https://api.acme.com/svc \
  --gcp-audience https://svc-abc-uc.a.run.app
```

Credentials are discovered through Google's Application Default Credentials (ADC) chain. The ambient `--gcp-id-token` path (no `--gcp-impersonate-service-account`) requires an ADC source whose credentials can mint ID tokens directly: service-account JSON keys, and the GCE / GKE / Cloud Run metadata server. External-account (Workload Identity Federation) and authorized-user (gcloud user creds) sources cannot mint ID tokens for arbitrary audiences on their own; pair them with `--gcp-impersonate-service-account`.

ID tokens are cached per `(impersonate_service_account, audience)` on the invoker (unbounded), and not cached on the admin/discovery path. Cached tokens are evicted 60 seconds before expiry. To reconfigure `auth` on an existing deployment, re-register with `--force`.

**Impact on Users**:
- Existing deployments are unaffected. The `auth` field is opt-in and defaults to absent.
- Lambda deployments are unaffected; the GCP flags are HTTP-only and both the REST handler and CLI reject the combination.
- Restate will not contact any Google API or read any ADC source until the first deployment with `auth` set is observed.

**Migration Guidance**: For self-hosted Restate calling private Cloud Run services, ensure the Restate process can reach Google's metadata server, or `GOOGLE_APPLICATION_CREDENTIALS` points to a readable service-account JSON key file. The minted token's principal must hold `roles/run.invoker` on the target Cloud Run service, and `roles/iam.serviceAccountOpenIdTokenCreator` on the impersonation target (if `impersonate_service_account` is used). For local development, run `gcloud auth application-default login` to populate ADC (distinct from `gcloud auth login`); authorized-user ADC credentials require `--gcp-impersonate-service-account`. Cloud Run services exposed via custom domains, ports, or traffic tags should set `--gcp-audience` explicitly.

_Related: #4755, #4448 (original multi-cloud auth surface)_

### Bounded Invoker Memory Pool

The invoker now operates under a **bounded memory pool** that limits the total amount of memory used for in-flight data flowing from the server to service deployments (the outbound path) — journal entries being replayed and service state being sent. Previously, this memory usage was unbounded and could grow without limit under high load or large payloads, risking OOM kills or degraded performance for all invocations on the node.

Key aspects:

- **Global memory pool**: a shared budget (`worker.invoker.memory-limit`, default **1.5 GiB**) limits total invoker memory across all partitions on a node.
- **Per-invocation budgets**: each invocation receives a budget capped at `worker.invoker.per-invocation-memory-limit` (defaults to `message-size-limit`, typically **32 MiB**).
- **Scheduler-level gating**: the scheduler only dispatches new invocations when the global pool has capacity for the initial reservation (`worker.invoker.per-invocation-initial-memory`, default **32 KiB**), preventing overcommitting.
- **Budget-aware storage reads**: journal replay and state loading acquire memory from the invocation's outbound budget before reading from storage, providing backpressure before data is materialized.

**Configuration**:

```toml
[worker.invoker]
# Global memory budget shared across all invocations on this node. Default: "1.5GB"
memory-limit = "1.5GB"

# Maximum memory a single invocation may use for its outbound data.
# Defaults to message-size-limit if unset. Clamped at message-size-limit.
per-invocation-memory-limit = "32MB"

# Memory reserved from the global pool before an invocation starts. Default: "32KB"
per-invocation-initial-memory = "32KB"
```

**Impact on Users**:
- **Most users will see no change**: defaults are generous; the 1.5 GiB pool supports thousands of concurrent normal-sized invocations.
- **Users processing very large payloads or state**: invocations exceeding the per-invocation limit (default 32 MiB) may now be paused with an "upper bound exceeded" error. Increase `per-invocation-memory-limit` if needed.
- **High-concurrency deployments**: if the global pool is exhausted, new invocations wait for memory rather than starting immediately. Increase `memory-limit` if throughput drops.

Note: `eager-state-size-limit` is automatically clamped to `per-invocation-memory-limit` since eager state is sent via the outbound budget.

_Related: #4354, #4345_

### HTTP/2 Connection Pool for Service Invocations

The invoker now uses a dedicated HTTP/2 connection pool when calling HTTP/2 service deployments. The pool keeps a set of multiplexed connections per `(scheme, authority)` and opens additional connections as concurrent stream demand grows, so the number of concurrent invocations to a deployment is no longer capped by a single connection's `max_concurrent_streams`. Capping per-connection streams also helps L4 load balancers spread traffic across backends instead of pinning it to one.

Connections idle past a configurable timeout are evicted automatically, and per-stream / per-connection flow-control windows are tunable for workloads with large payloads. The pool applies only to deployments registered as HTTP/2; HTTP/1.1 and ALPN-negotiated traffic are unaffected and services need no changes.

**Configuration** (under `[worker.invoker]`):

| Option                                  | Default | Description |
|-----------------------------------------|---------|-------------|
| `http2-streams-per-connection-limit`    | `128`   | Upper bound on per-connection concurrent HTTP/2 streams; caps the peer's advertised `max_concurrent_streams`. |
| `http2-idle-connection-timeout`         | `5m`    | How long a connection may stay idle before it is evicted. `0` disables eviction. |
| `http2-initial-stream-window-size`      | `2 MiB` | Initial flow-control window for received data on each stream (valid range 65535 B .. 2 GiB). |
| `http2-initial-connection-window-size`  | `5 MiB` | Initial connection-level flow-control window; should be ≥ the per-stream window (valid range 65535 B .. 2 GiB). |
| `http2-max-frame-size`                  | `16 KiB`| Largest HTTP/2 DATA frame payload the client will accept (valid range 16 KiB .. 16 MiB). |

**Impact on Users**:
- **New and existing deployments**: pick up the new behaviour automatically on upgrade; the defaults are safe for typical setups.
- **Migration**: none required. Tune the options above only to lower the per-connection stream cap when fronting an L4 load balancer, or to widen flow-control windows for large payloads.

### Partition Leadership Control

Operators can now control leader election for partition processors via `restatectl partition leader`. The policy is persisted in the metadata store and survives cluster controller restarts. It supports two independent controls:

- **Pin to node**: pin a partition's leader to a specific node (a soft preference — if the pinned node is unavailable, the scheduler falls back to automatic selection).
- **Election freeze**: pause leader election with a reason (a hard stop — the current leader stays, and no new leader is elected even if the current one dies).

```bash
restatectl partition leader pin 0-4 --node 2
restatectl partition leader unpin 0-4
restatectl partition leader freeze 0 --reason "Maintenance window"
restatectl partition leader unfreeze 0
restatectl partition leader show 0-4
```

**Impact on Users**: No action required for existing clusters — the default behavior (automatic leader selection) is unchanged.

_Related: #4576_

### Safe Ingress Retries

When the `controlled-idempotent-sharding` cluster feature is enabled (the default for newly provisioned clusters), the HTTP ingress automatically generates a random idempotency key for calls to services (`InvocationTargetType::Service`) and virtual objects (`InvocationTargetType::VirtualObject`) that do not already carry an `Idempotency-Key` header. Calls that already provide a key, and calls to workflows, are left untouched.

The motivation is **safe, transparent retries at the ingress**: with an idempotency key attached to every request, the ingress can unconditionally retry an in-flight invocation whenever its connection to the target partition processor is lost or the partition leader changes, without risking duplicate execution. Because the injected key is freshly randomized per request, no cross-request deduplication is introduced — only retries of the same in-flight request are collapsed.

The decision is one-way: once a cluster has accepted idempotent invocations under a given sharding scheme, changing it would re-shard those invocations and break deduplication — which is why the feature set is frozen in `NodesConfiguration` at provisioning time.

**Impact on Users**:
- **Clusters with `controlled-idempotent-sharding` enabled**: service and virtual object calls without an `Idempotency-Key` header are treated as idempotent invocations with a server-generated key, routed to one of the controlled buckets. As a side effect, the invocation's result is now retained after completion for the duration of the idempotency retention period.
- **Clusters with the feature disabled**, **workflows**, and **callers that already send an `Idempotency-Key`**: no change.

**Migration Guidance**: None required. Operators who do not want this behavior can opt out at provisioning time with `restatectl provision --disable-feature controlled-idempotent-sharding`.

#### Matching the previous cleanup behavior

The retention/cleanup window for these calls changes because the injected key makes them count as idempotent invocations. Previously, service and virtual object calls without an idempotency key were cleaned up according to the **journal retention** alone. With a key now present, the completion (the invocation's value) is retained for the **idempotency retention**, and the journal retention is capped so it never exceeds the idempotency retention.

So cleanup is now governed by *both* retention values rather than journal retention alone. Each value is taken from the per service/handler setting if one is configured (via the SDK / Admin API), otherwise from the server default:

- **Idempotency retention** — now drives how long the invocation metadata and value are retained for these calls. Server default: `default-idempotency-retention`, with an upper bound of `max-idempotency-retention`.
- **Journal retention** — still applies, but is capped at the idempotency retention. Server default: `default-journal-retention`, with an upper bound of `max-journal-retention`.

To get the exact previous cleanup window for these calls, set the idempotency retention to the journal retention you relied on before — either per service/handler (preferred) via the SDK / Admin API, or globally by setting `default-idempotency-retention` to match `default-journal-retention`.

_Related: #4662_

### Configurable Idempotency and Workflow Retention

`InvocationOptions` now exposes four new server-wide retention knobs alongside the existing `default-journal-retention` / `max-journal-retention` pair:

- `default-idempotency-retention`
- `max-idempotency-retention`
- `default-workflow-completion-retention`
- `max-workflow-completion-retention`

The `default-*` options provide the fallback retention used when neither the service nor the handler configures its own value. The `max-*` options act as a ceiling: any configured retention higher than the max is clamped down at read time (with a `SchemaInfo` message attached, same as journal retention). Previously, idempotency and workflow retention were stuck at a fixed 24-hour default with no max.

**Impact on Users**:
- **Defaults unchanged**: the new `default-*` options default to `24h`, matching prior behavior. Existing deployments see no retention difference on upgrade.
- **Storage semantics unchanged**: values are stored verbatim; clamping happens at read time.
- **New opt-in capability**: setting the `max-*` options starts clamping configured retentions that exceed your limit.

```toml
default-idempotency-retention = "1h"
max-idempotency-retention = "7d"
default-workflow-completion-retention = "30d"
max-workflow-completion-retention = "90d"
```

### Eager State Size Limit

A new configuration option `eager-state-size-limit` controls the amount of state sent eagerly to service endpoints in the `StartMessage`. When eager state is enabled (the default for Virtual Objects and Workflows), the server sends all state entries in the `StartMessage`; for services with large state, this forces the deployment to hold the entire state in memory. The new option caps the eager state size — when exceeded, the server sends partial state and the service fetches remaining keys lazily on demand, reducing memory pressure.

```toml
[worker.invoker]
# Maximum total size of state entries to send eagerly.
# Set to "0" to disable eager state entirely (equivalent to lazy state).
# Defaults to message-size-limit if unset, and is always clamped to message-size-limit.
eager-state-size-limit = "10MB"
```

**Impact on Users**:
- **Default behavior unchanged**: if unset, eager state is capped at the message size limit.
- **When the limit is set**: services with state exceeding the limit receive partial state and use `GetEagerState`/`GetEagerStateKeys` to fetch remaining keys lazily. All SDKs already support partial state — no changes required.
- **`disable_eager_state` users**: the option still works but is deprecated; migrate to `eager-state-size-limit = "0"`.

_Related: #4345, #4344 (longer-term streaming solution)_

### Configurable Request Identity Token Expiration

The validity window of the request-identity JWTs that the server attaches to outgoing service requests is now configurable via the new `worker.invoker.request-identity-expiration` option (default: `60s`). Previously this window was hard-coded to 60 seconds.

The value is applied symmetrically around the issuing time: the token's `exp` (expiry) is set to `now + expiration` and its `nbf` (not-before) to `now - expiration`, so it bounds both how long a token stays valid and how much clock skew between the server and the receiving SDK is tolerated. Deployments where the Restate server and SDK endpoints experience clock skew larger than the previously fixed 60-second leeway could see request-identity validation fail; operators can now widen (or tighten) the window to match their environment.

```toml
[worker.invoker]
request-identity-expiration = "60s"
```

**Impact on Users**:
- **Existing deployments**: no change in behavior; the default remains `60s`.
- **New deployments**: can set `request-identity-expiration` under `[worker.invoker]` to override the default. The option only has an effect when `request-identity-private-key-pem-file` is also configured.

### Filter Invocations by Completion Result

## Improvements

### Behavioral Changes

#### Handler Removal No Longer Requires the `breaking` Flag

Restate no longer rejects deployment registrations that remove handlers from an existing service revision. Previously, registering a new deployment that removed handlers would fail with error META0006 unless the `breaking` flag was set. Removing handlers is a valid operation during service evolution. The `breaking` flag now only controls whether service type changes are allowed.

_Related: #4543_

#### Services Tracing Revamp

The shape of services tracing has changed:

- The single long-running invocation span is gone. In its place, the partition processor emits three spans per invocation: one at start, one at the end, and one per attempt (emitted by the invoker).
- These spans are emitted **as soon as they're ready**, giving real-time feedback of running invocations in your tracing infrastructure rather than only at the end of the invocation.
- Per-command spans are gone. Commands are now recorded as events on the relevant attempt span.
- Spans created by Restate are now exported with the resource name `Restate` (instead of the service name), as the OTel spec requires.

Ingress spans are unchanged. Dashboards, alerts, or queries keyed on the old long-running invocation span name or on per-command spans will need to be updated.

_Related: #4530_

#### RocksDB Storage Tuning Defaults and New Knobs

Several RocksDB defaults were retuned for more predictable write throughput, and two new tuning knobs were added. Deployments using defaults need no action.

Default changes:

- `[log-server] rocksdb-blob-min-size`: raised from `4 KiB` to `512 KiB` (applies when blob separation is enabled).
- `[log-server] rocksdb-max-sub-compactions`: now defaults to `1`. The previous default `0` — which mapped to `floor(CPU / 2)` — is no longer special-cased; set an explicit value to enable sub-compactions.
- `[worker.storage] rocksdb-max-background-flushes`: default lowered from `2` to `1`.
- `[worker.storage] rocksdb-max-background-compactions`: default lowered from `2` to `1`.

New options:

- `[log-server] rocksdb-wal-dir`: optional path to a dedicated directory for WAL files (defaults to the log-store data directory). Point it at a low-latency NVMe SSD with sufficient IOPS/bandwidth, especially when `rocksdb-disable-wal-fsync` is `false`.
- `[log-server] rocksdb-max-file-size`: clamps the target SST file size (default `128 MiB`; sanitized up to a `16 MiB` minimum).
- `[worker.storage] rocksdb-max-file-size`: clamps the target SST file size (default `64 MiB`; sanitized up to a `16 MiB` minimum).

RocksDB Prometheus export now includes additional RocksDB statistics for block-cache activity, merge operators, flush payload/garbage bytes, get efficiency, compaction, compression, read amplification, write stalls, and total SST file size. RocksDB perf-context metrics also cover more read/write internals such as index/filter/data-block reads, bloom-filter hits/misses, block-cache hits, merge-operator time, RocksDB mutex time, and write-delay time.

The `restate.invoker.available_slots` metric has been dropped and replaced by two cumulative per-node counters: `restate.invoker.concurrency_slots.acquired` and `restate.invoker.concurrency_slots.released`. These make it easy to derive the slot acquisition/release rate and node-level available slots:

```promql
sum by (node_name) (restate_invoker_concurrency_limit)
  - (
    sum by (node_name) (restate_invoker_concurrency_slots_acquired_total)
    - sum by (node_name) (restate_invoker_concurrency_slots_released_total)
```

> **Future breaking observability change**: In Restate v1.8.0, the `invoker_id` label will be removed from `restate.invoker.concurrency_limit`, so this metric will always be reported at node scope. Update dashboards, alerts, and recording rules to group by the node label instead of `invoker_id`.

`restate.invoker.task_duration.seconds` is no longer emitted with a `partition_id` label. Remove `partition_id` selectors from queries against this histogram.

The `partition_state` table gains an `enabled_features` column (`List<Utf8>`) listing the state-machine features active on each partition processor; the same is surfaced as the `FEATURES` column in `restatectl partition list`. Feature names are carried as strings (rather than typed booleans) so older `restatectl` clients can render features they don't recognise.

```sql
-- Which partitions have vqueues enabled?
SELECT partition_id FROM partition_state WHERE array_has(enabled_features, 'vqueues');
```

#### Partition State: Applied Rule-Book and Schema Versions

`partition_state` now exposes `applied_rule_book_version` (set once the first `UpsertRuleBook` command has been observed) and `applied_schema_version`, also surfaced as the `RULE-BOOK` and `SCHEMA` columns in `restatectl partition list`.

#### New Invoker and Ingress Traffic Metrics

Several additive Prometheus metrics give better visibility into traffic between the server, deployments, and ingress:

- **Invoker egress** — byte and request counters for data flowing between the invoker and service deployments (e.g. `restate.invoker.sent.bytes.total`, `restate.invoker.received.bytes.total`, `restate.invoker.client_requests.total`).
- **HTTP ingress connections** — connection lifecycle counters `restate.ingress.http.connection_created.total` and `restate.ingress.http.connection_dropped.total` (the latter carrying a `status` label that distinguishes clean closes from errors).
- **Ingress latency breakdown** — the ingress request-latency histogram now carries an `rpc.type` label (`call` vs `send`) so the two call shapes can be analyzed separately.

RocksDB now exports flush/compaction event metrics (flush- and compaction-completed counters, compaction duration, memtable-sealed counter) in addition to the perf-context statistics above. The partition store gained gauges and counters such as its memtable budget in bytes, the number of open partitions, and reclaim-flush counts. All are additive and require no configuration or dashboard migration.

### CLI Improvements

#### `restate state edit` Retries on Parse Error

`restate state edit` now re-opens the editor when the edited JSON fails to parse, instead of discarding your work. In interactive use you'll see the parse error and a prompt to re-open the editor; non-interactive use (`--yes` / CI) is unchanged and fails immediately on invalid JSON.

### Notable Internal Changes

#### Idempotency Table Removed

The legacy idempotency table has been removed from the partition store along with all related code. Restate stopped writing to this table in v1.3 — invocation deduplication has since used the deterministic invocation id derived from the idempotency key — and by v1.7 any entries written before v1.3 have aged well past their retention window.

There is **no user-visible behavior change**: idempotent invocations continue to work via the deterministic invocation id. However, the `sys_idempotency` SQL table is no longer available through the storage query interface, so any tooling that selected from it must be updated. The `KeyKind::Idempotency` byte prefix (`b"ip"`) stays reserved, so storage data files written by v1.2 or older that still contain idempotency entries will not block startup (those entries are simply no longer consulted).

_Related: #4536_

## Bug Fixes

### Awakeable Signal Loss During Leadership Transitions

Fixed a bug where awakeable resolutions and signal deliveries could be silently lost during partition leadership transitions. When a leadership change occurred, the old leader's self-proposed commands (e.g., `AppendSignal` for awakeable resolution) could be committed to the Bifrost log after the new leader's `AnnounceLeader` entry, and the dedup mechanism then dropped them as outdated. Signal and invocation-response commands from ingress RPCs are now appended without dedup information, so they are never filtered during leadership transitions (the SDK handles potential duplicate signals gracefully). No action required.

_Related: #4566_

### Deterministic Random Seeds for Restart-From-Prefix

Restart-from-prefix could replay copied prefix entries against a different random seed than the original execution used, because the seed exposed by `ctx.rand()` was derived lazily from the invocation id — and restart-from-prefix mints a new invocation id. Any handler that consumed `ctx.rand()` inside the copied prefix could observe non-deterministic replay or replay errors.

The fix persists a unique random seed on every new invocation and carries it over when restart-from-prefix replays existing entries. This is **opt-in** in v1.7 and is a backwards-incompatible change: once enabled, rollback to a server version < v1.7.0 is no longer possible. Anyone using (or planning to use) restart-from-prefix on services that call `ctx.rand()` should opt in; the default flips in v1.8.0.

```toml
experimental_enable_unique_random_seeds = true
```

Note: invocations created before activation keep deriving the seed from the invocation id (no backfill); only invocations created after activation get a persisted seed.

_Related: #4750_

### Orphaned Journal Index Entries Cleanup

Fixed a bug where journal index entries were not cleaned up when journals were deleted, causing them to accumulate over time and waste disk space. A one-time background cleanup task now runs automatically on partition processor startup to remove orphaned entries; it runs in the partition processor runtime without blocking partition processing, is tracked per partition store, runs once, and retries on the next startup if interrupted. No action required.

_Related: #4519_

### CLI: `--remote-port` in `restate cloud env tunnel`

`--remote-port` was completely broken — it rejected all input and displayed `[possible values: Self::DEFAULT_PORT, Self::DEFAULT_PORT]`. It now correctly accepts `8080` (ingress) and `9070` (admin). A new `--no-remote-ports` flag disables environment port proxying while keeping only the inbound tunnel direction.
